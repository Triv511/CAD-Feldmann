Git Workflows

Discover proven development strategies and branching models that scale with your team

Popular Git Workflows

üåø Feature Branch Workflow

The most popular workflow where each feature is developed in its own branch before being merged into main. Perfect for small to medium teams.

git checkout -b feature/new-login
git merge feature/new-login

üîÑ GitFlow Workflow

A robust branching model with dedicated branches for features, releases, and hotfixes. Great for projects with scheduled releases.

git flow feature start new-feature
git flow feature finish new-feature

üöÄ GitHub Flow

A simplified workflow focused on continuous deployment. Features are developed in branches and deployed via pull requests.

git checkout -b feature-branch
git push origin feature-branch

Feature Branch Workflow Deep Dive

Step-by-Step Process

1. üÜï Start New Feature

git checkout main
git pull origin main
git checkout -b feature/user-authentication

Always start from the latest main branch to avoid conflicts.

2. üíª Develop Feature

git add .
git commit -m "Add login form"
git push origin feature/user-authentication

Make regular commits and push to remote for backup.

3. üîÄ Merge Feature

git checkout main
git pull origin main
git merge feature/user-authentication
git push origin main

Update main, merge feature, and push changes.

GitFlow Branching Model

üè† Main Branches

Master/Main Branch

Production-ready code. Every commit represents a potential release.

Develop Branch

Integration branch where features come together. Next release preparation.

üåø Supporting Branches

Feature Branches

Branch from: develop
Merge back to: develop
Naming: feature/*

Release Branches

Branch from: develop
Merge to: main and develop
Naming: release/*

üö® Hotfix Branches

Critical Fixes

Branch from: main
Merge to: main and develop
Naming: hotfix/*

git checkout -b hotfix/critical-bug main

For urgent production fixes that can't wait for next release.

GitHub Flow - Simplified Process

üìã Pull Request Process

  1. Create feature branch from main
  2. Make commits and push to remote
  3. Open pull request for review
  4. Discuss and review code changes
  5. Deploy branch for testing
  6. Merge to main after approval

üöÄ Continuous Deployment

Main branch is always deployable. Every merge to main can trigger automated deployment to production.

git checkout -b add-dark-mode
# Make changes, commit, push
# Create PR, review, merge

üë• Code Review Culture

Every change goes through peer review via pull requests. This ensures code quality and knowledge sharing across the team.

üë• Small Team (2-5 developers)

üè¢ Large Team (10+ developers)

Recommended: GitFlow

Advanced Workflow Techniques

üîÑ Rebase vs Merge

git rebase main
git merge --no-ff feature-branch

Rebase creates linear history, merge preserves context. Choose based on your team's preference.

üçí Cherry Picking

git cherry-pick [commit-hash]

Apply specific commits from one branch to another. Useful for hotfixes or selective feature porting.

üì¶ Squash Commits

git rebase -i HEAD~3

Combine multiple commits into one for cleaner history before merging to main branch.
